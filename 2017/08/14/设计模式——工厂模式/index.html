设计模式——工厂模式

2017-08-14
  • 创建型模式

创建型模式

在GOF23中，属于创建型模式的有：

  1. 单例模式
  2. 工厂模式
  3. 抽象工厂模式
  4. 建造者模式
  5. 原型模式

工厂模式正是其中一种。工厂方法模式其实也可以分为1、简单工厂模式，2、工厂方法模式。在设计模式一书中，将简单工厂模式当作了工厂方法模式的一种特例，所以，通常所说的工厂模式就是工厂方法模式。

简单工厂模式

简介/动机

涉及到的编程原则

开闭原则，依赖倒转原则。（不清楚的可以查看设计模式——面向对象的基本原则一文）

动机

隐藏对象的创建过程，直接为客户端提供一个符合要求的实例对象（这个对象属于一个系列中的一种）。

解决场景

动机部分说的不是很具体
1、场景一：我们试想一下，通常我们创建对象时，通过new就可以了。但是，如果我们有一系列对象，因为性质相同，我们将这一系列产品抽象为接口。根据依赖倒转原则，我们是不能直接new的，需要根据接口来获得具体的产品对象，那么，具体该怎么做呢？这就需要我们额外提供一中方式，位客户端提供一个傻瓜式方式来获取对象，那就是工厂方法模式。
2、场景二：我们需要获取的目标对象，只是某个业务中的一部分，在它创建之前，需要设置一些其他属性，或者初始化别的对象。就像是一个系统，我们的目标对象孤立存在并没有现实意义，这个时候我们让客户端单独创建目标对象，同时还让他将关联的其它业务细节都处理一遍，他一定会杀了你。怎么解决，工厂模式来告诉你。

角色

简单工厂的角色比较简单
1、 一个抽象产品类，可以派生一系列产品。
2、 一个工厂类，用于生成具体产品。

举个栗子

任务

生产一系列的衣服

步骤1：产品抽象接口和具体产品对象

public interface Clothes {
    void product();
}

public class Nike implements Clothes{

    @Override
    public void product() {

        System.out.println("Nike --> producted");
    }

}

public class Adidas implements Clothes{

    @Override
    public void product() {

        System.out.println("Adidas --> producted");
    }

}

步骤2：工厂类

public class ClothesFactory {
    public static Clothes create(String type) {
        if ("adidas".equals(type)) {
            return new Adidas();
        } else if ("nike".equals(type)) {
            return new Nike();
        } else {
            return null;
        }
    }
}

步骤3：调用

public class Client02 {
    public static void main(String[] args) {
        Clothes adidas = ClothesFactory.create("adidas");
        Clothes nike = ClothesFactory.create("nike");
        nike.product();
        adidas.product();
    }
}

结语

从类图关系中，我们可以看出，客户端只需要告诉工厂需要什么品牌的衣服，就能拿到衣服。不仅不需要衣服的产生细节，同时在也需要知道具体类Adidas或者Nike，使调用结构更加简单。

但是，问题来了，如果我现在想新增一个产品，比如说Erke(鸿星尔克)。至少有两个步骤
step1: 实现抽象产品类
step2: 修改工厂方法，新增一个case来满足需求。像这样

public class ClothesFactory {
    public static Clothes create(String type) {
        if ("adidas".equals(type)) {
            return new Adidas();
        } else if ("nike".equals(type)) {
            return new Nike();
        } else if ("erke".equals(type)){//新增case
            return new Erke();
        }
    }
}

很明显，step2违反了开闭原则。

工厂方法模式，提供了简单工厂问题的解决方案。

工厂模式

简介/动机

解决简单工厂带来的违反开闭原则的问题。

角色

1、抽象产品类，可以派生一系列产品。
2、抽象工厂类，可以派生一系列共产。
3、具体产品类，和具体工厂类。一个产品对应一个工厂类，工厂类只生产对应的产品。

栗子

依然是生产衣服

步骤1:

将上一个栗子的工厂方法抽象为一个接口，同时为每中衣服提供一个该接口的实现，来生产具体类。

public interface ClothesFactory {
    Clothes createClothes();
}

public class AdidasFactory implements ClothesFactory{

    @Override
    public Clothes createClothes() {
        return new Adidas();
    }

}

public class NikeFactory implements ClothesFactory {

    @Override
    public Clothes createClothes() {

        return new Nike();
    }

}

步骤2：调用

public class Client03 {
    public static void main(String[] args) {
        Clothes adidas = new AdidasFactory().createClothes();
        Clothes nike = new NikeFactory().createClothes();
        adidas.product();
        nike.product();
    }
}

类图关系

分析

工厂方法模式，为每一个具体工厂负责生产对应的具体产品类。这样，不管以后新增多少具体产品，我们都加上对应的具体工厂类就可以了（在简单工厂的基础上，满足了开闭原则）。

真正的结语

具体应该选择什么方式，就要看具体业务了。如果产品可以遇见不会很多，简单工厂也是OK的，毕竟工厂模式会新增大量的工厂类，增加了项目结构的复杂度，维护性也会降低。见仁见智了。
